## 실행 절차

### Step 1: 날짜 계산

오늘 날짜(KST)를 기준으로:
- **오늘**: 현재 날짜 (YYYY-MM-DD)
- **어제**: 전 영업일 계산
  - 월요일이면 → 직전 금요일
  - 그 외 평일이면 → 전날

### Step 2: 내 프로젝트 조회

`notion-query-data-sources`로 전체 프로젝트 리스트 DB를 쿼리한다:

```sql
SELECT url, "포지션", "Driver"
FROM "collection://0ffbed16-cd0a-4381-b293-b3e630f1d06d"
WHERE "프로젝트 분류" = '진행중인 이니셔티브'
```

결과에서 `Driver` 필드에 `310d872b-594c-812e-b436-0002c6a46fa9`가 포함된 행만 필터링한다.
각 프로젝트의 "포지션"(프로젝트명)과 URL을 저장한다.

### Step 3: 스크럼 메시지 찾기

`slack_read_channel`로 #스크럼 채널(`C09TF3XFJQ0`)의 최근 메시지를 조회한다 (limit 20).

필터링 조건:
- 발신자가 rbot(`U0AJT72CFLP`)인 메시지
- 당일 오전 9:25~9:35 KST 범위에 발송된 메시지
- 메시지 본문에 `Daily Scrum`이 포함된 메시지

각 메시지에서 프로젝트명을 추출한다 (볼드 텍스트의 두 번째 줄, `*프로젝트명*` 형식).

### Step 4: 프로젝트-메시지 매칭

Step 2에서 조회한 프로젝트명(포지션)과 Step 3에서 추출한 메시지 제목을 비교한다.
- 부분 문자열 매칭: 프로젝트명이 메시지 제목에 포함되거나, 메시지 제목이 프로젝트명에 포함되는 경우 매칭
- 매칭된 메시지의 `message_ts`를 저장한다

매칭되지 않은 프로젝트는 스킵한다 (해당 프로젝트의 스크럼 메시지가 오늘 생성되지 않은 경우).

### Step 5: Daily DB 정보 수집 (핵심 데이터 소스)

`notion-query-data-sources`로 오늘과 어제의 Daily 항목을 조회한다:

```sql
SELECT url, "Name", "date:Date:start"
FROM "collection://31119aac-27e4-8008-bb4b-000bd838f764"
WHERE "date:Date:start" IN ('{오늘}', '{어제}')
ORDER BY "date:Date:start" DESC
```

각 Daily 페이지를 `notion-fetch`로 개별 조회하여 내용을 수집한다:

**어제 Daily 페이지에서:**
- `[x]` 완료된 항목 → "어제 완료" 목록으로 분류
- `[ ]` 미완료 항목 → 이월 항목으로 참고

**오늘 Daily 페이지에서:**
- `[ ]` 미완료 항목 → "오늘 진행" 목록으로 분류
- `[x]` 이미 완료된 항목 → 포함 가능

Daily 항목 내에 프로젝트별 섹션 헤더가 있으면, 해당 프로젝트에 매핑한다.

**Daily DB 포맷 (SSoT 구조):**
Daily 페이지는 `## todo` 섹션 아래에 이니셔티브별로 구조화되어 있다:
```
## todo
* [이니셔티브명]
	- [x] 완료 항목 (1탭 들여쓰기 = 하위 요소)
	- [ ] 미완료 항목
* [다른 이니셔티브명]
	- [x] 완료 항목
### misc.
- 이니셔티브에 속하지 않는 기타 업무
```

`* [이니셔티브명]`은 최상위 bullet이고, 그 아래 체크박스들은 탭 들여쓰기된 하위 요소이다.
`* [이니셔티브명]` 헤더와 Step 2에서 조회한 프로젝트명(포지션)을 매칭한다.
부분 문자열 매칭: 이니셔티브 태그가 프로젝트명에 포함되거나, 프로젝트명이 이니셔티브 태그에 포함되는 경우 매칭한다.

레거시 포맷(태그 없이 `- 프로젝트명` 형태)도 하위 호환으로 지원한다.
`### misc.` 섹션의 항목도 관련성이 있으면 포함한다.

**서브페이지 추가 조회:**
각 Daily 페이지의 `notion-fetch` 결과에서 `<page url="...">` 태그가 있으면,
해당 URL로 `notion-fetch`를 추가 호출하여 서브페이지 내용도 수집한다.
서브페이지의 체크리스트 항목(`[x]`, `[ ]`)도 부모 페이지와 동일한 기준으로 분류한다.
단, 서브페이지가 3개를 초과하면 제목 기준으로 프로젝트 관련성이 높은 것만 선별하여 fetch한다.

### Step 6: 프로젝트 페이지 보조 참고 (fallback)

Daily DB에 특정 프로젝트 관련 항목이 **부족한 경우에만** 해당 프로젝트 페이지를 `notion-fetch`로 조회한다.

프로젝트 페이지에서 참고할 섹션:
- "🔧 진행 중 / 개선 필요" → 오늘 진행 후보
- "✅ 구현 완료" 섹션의 최근 항목 → 어제 완료 후보
- Objective 섹션의 `[ ]` 미완료 항목 → 오늘 진행 후보

**주의**: 프로젝트 페이지는 전체 로드맵이므로 날짜 구분이 불가하다. Daily DB가 항상 우선이며, 프로젝트 페이지는 보조 참고용이다.

**서브페이지 추가 조회:**
프로젝트 페이지의 `notion-fetch` 결과에서 `<page url="...">` 태그가 있으면,
관련성이 높은 서브페이지(진행 중/최근 업데이트된 항목)를 `notion-fetch`로 추가 조회한다.
서브페이지 내용에서 체크리스트 항목이나 진행 상황을 보조 데이터로 활용한다.
단, 프로젝트 페이지의 서브페이지는 로드맵 전체를 포함할 수 있으므로, 최대 2개까지만 fetch한다.

### Step 7: 스크럼 메시지 작성

매칭된 각 프로젝트별로 메시지를 구성한다. 형식:

```
어제 완료:
• {완료 항목1}
• {완료 항목2}

오늘 진행:
• {진행 항목1}
• {진행 항목2}
```

**태그 사용 조건** (해당되는 경우에만 추가):
- `이슈:` → 진행이 멈추거나 해결하지 못한 항목이 명확히 있을 때만 사용
- `도움 필요:` → Daily/프로젝트 페이지에서 누군가를 태그했거나, 누군가의 확인이 필요함을 명시한 경우만 사용
- `확인 필요:` → 위와 동일한 조건

태그가 불필요하면 사용하지 않는다. 억지로 만들지 않는다.

**작성 원칙:**
- 간결하게, 핵심만 기술
- 기술 용어/프로젝트 약어는 그대로 사용
- 과거 스레드 답변과 비슷한 톤 유지
- 불확실한 내용은 포함하지 않음

### Step 8: Slack DM으로 초안 전송

`slack_send_message`로 김하영 본인 DM(`U0AGKLZUCSY`)에 초안을 전송한다.

메시지 형식:
```
:memo: *Daily Scrum 초안* ({오늘 날짜})

*{프로젝트1명}*
스레드: {스레드 퍼머링크}

{프로젝트1 스크럼 메시지}

*{프로젝트2명}*
스레드: {스레드 퍼머링크}

{프로젝트2 스크럼 메시지}
```

스레드 퍼머링크 형식: `https://rebuilderai.slack.com/archives/C09TF3XFJQ0/p{message_ts에서 점 제거}`

### Step 9: 결과 보고

작성된 초안 내용과 대상 스레드 정보를 사용자에게 요약하여 알린다.
사용자가 DM에서 내용 확인 후 각 스레드에 복사-붙여넣기로 전송한다.

## 주의사항

- **Slack MCP와 Notion MCP 연결이 모두 필요하다.** 연결이 안 되어 있으면 사용자에게 안내한다.
- 초안은 DM으로만 전송하고, **직접 스레드에 전송하지 않는다.**
- 매칭되는 스크럼 메시지가 없으면 "오늘은 담당 프로젝트의 스크럼 메시지가 아직 없습니다"라고 알린다.
- Daily DB에 오늘/어제 항목이 없을 수 있다. 이 경우 프로젝트 페이지 fallback을 사용한다.
- 오늘이 월요일이면 "어제"는 금요일이다.
- Slack 메시지에서 em dash, 화살표 등 특수문자는 -, -> 등 ASCII로 대체한다. 특수문자가 invalid_blocks 에러를 유발할 수 있다.
- DM 메시지 본문에는 --- 구분선을 사용하지 않고, 빈 줄로 프로젝트 간 구분한다.
- 스레드 링크는 DM 본문이 아닌 스레드 답장으로 별도 전송한다.
- Slack MCP를 사용할 수 없는 경우, 스크럼 초안을 텍스트로 출력하고 "Slack MCP 연결 불가. 아래 초안을 직접 Slack에 붙여넣으세요."라고 안내한다.
