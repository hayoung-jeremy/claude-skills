# Wrap-Up: 작업 내용 노션 정리 스킬

## 역할

사용자의 작업 내용을 분석하고, 지정된 노션 DB에 정리해요.

## 핵심 ID 참조

| 항목 | ID |
|------|-----|
| Daily DB | `collection://31119aac-27e4-8008-bb4b-000bd838f764` |
| Notes DB | `collection://31119aac-27e4-80d9-a83c-000be57af066` |
| 전체 프로젝트 DB | `collection://0ffbed16-cd0a-4381-b293-b3e630f1d06d` |
| 통합 태스크 DB | `collection://db1ed8bf-4edf-4a87-b5bd-8774ce361a73` |
| 김하영 Notion User ID | `310d872b-594c-812e-b436-0002c6a46fa9` |

## 0단계: 기록 대상 DB 결정

사용자의 요청에서 **어느 DB에 기록할지** 판별해요.

| 사용자 표현 | 대상 DB |
|------------|---------|
| "daily에 정리해", "데일리에 적어줘" | Daily DB |
| "notes에 정리해", "노트에 적어줘", "노트로 작성해줘" | Notes DB |
| "정리해줘", "wrap up", "노션에 정리해줘" (DB 미지정) | **사용자에게 질문** |

**DB가 명시되지 않은 경우**, 반드시 사용자에게 질문해요:
> "Daily와 Notes 중 어디에 기록할까요?"

사용자가 응답할 때까지 기록을 진행하지 않아요.

## 1단계: 정리 대상 파악

아래 우선순위로 정리 대상을 결정해요:

1. **사용자가 명시적으로 지정한 경우** - 해당 대상만 정리
   - 예: "feat/2.5d-edit 브랜치 작업 정리해줘" -> 해당 브랜치 분석
   - 예: "오늘 커밋 정리해줘" -> 오늘 날짜 커밋만 분석
   - 예: "이번 대화 내용 정리해줘" -> 대화 컨텍스트 기준
2. **$ARGUMENTS가 있는 경우** - 해당 내용 기준으로 분석
3. **아무 지정 없는 경우** - 현재 대화 컨텍스트에서 수행한 작업 기준

### 대화 시간대 파악

대화에서 수행된 작업의 시간대를 파악해요. 커밋 시간, 대화 흐름 등을 기반으로 "오전/오후/저녁" 등의 맥락을 반영해요.

### 브랜치/커밋 분석이 필요한 경우

- 현재 브랜치 확인: `git branch --show-current`
- 리모트 브랜치라면: `git fetch origin <branch>` 먼저 실행
- 오늘 커밋 필터: `git log --since="today" --format="%h %ad %s" --date=short`
- 해당 브랜치 고유 커밋만 추출 (main/prod 대비 diff)
- 변경 파일 확인: `git diff <base>...<target> --stat`
- 핵심 파일 내용은 `git show <branch>:<filepath>`로 읽어요 (체크아웃 불필요)

## 2단계: 태스크/프로젝트 매칭

작업 내용이 어떤 태스크(프로젝트)에 해당하는지 판별해요.

### 내 태스크/프로젝트 조회

2단계로 조회해요:

**1) 통합 태스크 DB에서 내가 담당하는 진행중 태스크 조회:**

```sql
SELECT url, "태스크명", "프로젝트", "상태"
FROM "collection://db1ed8bf-4edf-4a87-b5bd-8774ce361a73"
WHERE "상태" NOT IN ('완료', '아카이빙')
```

결과에서 `담당자` 필드에 `310d872b-594c-812e-b436-0002c6a46fa9`가 포함된 행만 필터링해요.
(담당자 필드는 `["user://310d872b-..."]` JSON 배열 형식이므로 문자열 포함 여부로 판별)

각 태스크의 `프로젝트` relation URL로 프로젝트 페이지를 `notion-fetch`하여 `프로젝트명`을 추출해요.

**2) 프로젝트 DB에서 내가 담당자(PL)인 프로젝트도 조회:**

```sql
SELECT url, "프로젝트명", "담당자(PL)"
FROM "collection://0ffbed16-cd0a-4381-b293-b3e630f1d06d"
WHERE "상태" = '[진행중] 프로젝트'
```

결과에서 `담당자(PL)` 필드에 `310d872b-594c-812e-b436-0002c6a46fa9`가 포함된 행만 필터링해요.

### 매칭 규칙

- 대화에서 다룬 작업의 키워드(repo 이름, 브랜치명, 기술 스택)와 태스크명을 비교해요
- 매칭되는 태스크가 있으면 해당 태스크명을 사용해요
- 매칭되지 않는 작업은 misc로 분류해요

## 3단계: 지정된 DB에 기록

### Daily DB에 기록하는 경우

오늘 날짜의 Daily 페이지에 구조화된 항목을 추가해요.

#### 오늘 Daily 페이지 찾기

```sql
SELECT url, "Name", "date:Date:start"
FROM "collection://31119aac-27e4-8008-bb4b-000bd838f764"
WHERE "date:Date:start" = '{오늘}'
```

- 페이지가 있으면 -> `notion-fetch`로 현재 내용을 확인하고 append
- 페이지가 없으면 -> 새 Daily 페이지를 생성

#### Daily 기록 포맷

Daily 페이지의 `## todo` 섹션에 아래 형식으로 기록해요:

```markdown
## todo
* [태스크명]
	- [x] 완료한 작업 항목1
	- [x] 완료한 작업 항목2
	- [ ] 아직 안 끝난 작업
		- [x] 세부 항목 (완료됨)
		- [ ] 세부 항목 (미완료)
* [다른 태스크명]
	- [x] 완료한 작업
	- [ ] 진행할 작업
### misc.
- 태스크에 속하지 않는 기타 업무/메모
```

#### Notion 마크다운 작성 시 주의사항

**들여쓰기가 핵심이에요.** Notion에서 하위 항목은 반드시 탭(`\t`) 들여쓰기로 부모-자식 관계를 표현해야 해요.
- `* [태스크명]`은 최상위 bullet이에요
- 그 아래 체크박스 항목들은 **반드시 1탭 들여쓰기**해서 하위 요소로 만들어야 해요
- 세부 항목은 **2탭 들여쓰기**해요
- 같은 레벨(들여쓰기 없음)에 넣으면 형제 요소가 되어 구조가 깨져요

#### 기록 규칙

- **태스크별로 그룹핑**: `* [태스크명]` 헤더 아래에 해당 태스크 항목을 하위 요소로 모아요
- **체크박스 사용**: 완료된 작업은 `- [x]`, 미완료/진행중은 `- [ ]`
- **기존 내용 보존**: 이미 적혀 있는 항목은 수정하지 않고, 새 항목만 추가해요
- **중복 방지**: 이미 기록된 동일 내용이 있으면 추가하지 않아요
- 태스크에 해당하지 않는 작업은 `### misc.` 섹션에 일반 bullet으로 기록해요

#### append 방법

기존 Daily 페이지에 이미 `## todo` 섹션이 있는 경우:
1. `notion-fetch`로 현재 페이지 내용을 읽어요
2. 이미 해당 태스크 섹션(`* [태스크명]`)이 있으면 그 아래에 항목 추가
3. 해당 태스크 섹션이 없으면 새 섹션을 추가
4. `notion-update-page` command: `append_content`로 내용을 추가해요

### Notes DB에 기록하는 경우

Notes DB에 상세 페이지를 생성해요.

#### Notes 문서 구조

```markdown
## 요약
(1~3줄로 작업 전체 요약)

## 작업 내용
- [x] 완료된 작업 항목
- [ ] 미완료 작업 항목

## 상세 노트
(기술적 설명, 파일 구조, 디버깅 이력 등)
```

#### Notes 생성 방법

```
notion-create-pages:
  parent: { data_source_id: "31119aac-27e4-80d9-a83c-000be57af066" }
  pages: [{ properties: { "Name": "<정리 제목>" }, content: "<마크다운 본문>" }]
```

#### 사용자가 URL을 명시한 경우

사용자가 노션 URL을 직접 제공하면 해당 위치에 생성해요:
- `notion-fetch`로 URL의 타입을 확인해요
- 데이터베이스 -> 해당 DB의 data_source_id로 페이지 생성
- 일반 페이지 -> 해당 페이지 하위에 생성

## 4단계: 결과 반환

실제 기록한 DB에 대해서만 결과를 보여줘요.

Daily DB에 기록한 경우:
```
정리 완료했어요!

Daily 업데이트:
- [Create/Variation 모델 학습 및 검증] 3개 항목 추가 (완료 2, 진행중 1)
```

Notes DB에 기록한 경우:
```
정리 완료했어요!

Notes 생성:
- UV 맵핑 알고리즘 비교 분석 (https://notion.so/xxx)
```

## 주의사항

- 노션 MCP 연결이 안 되어 있으면 사용자에게 연결 필요를 안내해요
- 페이지 접근 권한 오류 시 Connection 추가 필요를 안내해요
- Daily DB의 기존 내용은 절대 삭제하지 않아요 - append만 해요
- 사용자 확인 없이 민감한 정보(API 키, 비밀번호 등)를 노션에 작성하지 않아요
- Notes 작성 시 아이콘을 "claude" 커스텀 이모지로 설정하려 시도해요. 불가하면 스킵해요
