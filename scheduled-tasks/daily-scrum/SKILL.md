**에러 유형별 대응:**

| 에러 유형 | 대응 | DM 보고 |
|-----------|------|---------|
| Notion MCP 타임아웃 | 최대 2회 재시도 후 실패 시 다음 단계로 | O - 어떤 호출이 타임아웃 났는지 명시 |

| Notion 페이지 fetch 실패 | 해당 페이지 스킵, 나머지 진행 | O - 실패한 페이지 URL 명시 |
| Slack MCP 연결 불가/토큰 만료 | 즉시 중단 (DM 전송 자체 불가) | 텍스트 출력으로 대체 |
| Slack 메시지 전송 실패 (invalid_blocks 등) | 특수문자 제거 후 재시도 1회 | O - 원본 메시지와 에러 원문 명시 |
| Slack 채널 읽기 실패 | 스크럼 스레드 매칭 없이 진행 | O - 채널 ID와 에러 명시 |
| GitHub CLI 미인증/실패 | 해당 단계 스킵 | O - gh 에러 메시지 명시 |
| 캐시 파일 읽기/쓰기 실패 | 캐시 없이 진행 (읽기) / 스킵 (쓰기) | O - 파일 경로와 에러 명시 |
| 프로젝트-메시지 매칭 0건 | 정상 케이스일 수 있음 (rbot 미생성) | O - "오늘 담당 프로젝트 스크럼 스레드 없음" |
| Daily DB 조회 결과 0건 | fallback(캐시 Diff + GitHub)으로 진행 | O - "Daily DB에 {날짜} 항목 없음" |
| 통합 태스크 DB 조회 실패 | 전체 프로젝트 DB fallback 사용 | O - 에러 내용 명시 |
| 예상치 못한 에러 | 즉시 DM 보고 후 중단 | O - 에러 전문(traceback 포함) |

**핵심 원칙: 어떤 에러든 조용히 실패하지 않는다. 반드시 DM으로 보고한다. Slack DM 전송 자체가 불가능한 경우만 예외이다.**

## 실행 절차

### Step 1: 날짜 계산

오늘 날짜(KST)를 기준으로 조회 기간을 계산한다:
- **오늘**: 현재 날짜 (YYYY-MM-DD)
- **조회 시작일**: 직전 영업일
  - 월요일이면 → 직전 금요일
  - 그 외 평일이면 → 전날
- **조회 기간**: 조회 시작일부터 어제까지의 모든 날짜
  - 예: 월요일(3/16) 스크럼 → 금(3/13), 토(3/14), 일(3/15) 3일치
  - 예: 화요일 스크럼 → 월요일 1일치

### Step 2: 내 태스크 및 프로젝트 조회

**2-1) 통합 태스크 DB 조회 (주 경로)**

`notion-query-data-sources`로 통합 태스크 DB에서 내가 담당하는 활성 태스크를 조회한다:

```sql
SELECT url, "태스크명", "상태", "담당자", "프로젝트", "단계"
FROM "collection://db1ed8bf-4edf-4a87-b5bd-8774ce361a73"
WHERE "상태" NOT IN ('완료', '아카이빙')
```

결과에서 `담당자` 필드에 `310d872b-594c-812e-b436-0002c6a46fa9`가 포함된 행만 필터링한다.
(담당자 필드는 `["user://310d872b-..."]` JSON 배열 형식이므로 문자열 포함 여부로 판별)

각 태스크에서 다음 정보를 추출한다:
- `태스크명`: 태스크 제목. **Daily 태그와 매칭에 사용**
- `프로젝트`: relation 필드 (전체 프로젝트 DB의 페이지 URL 배열). **스크럼 스레드 매칭에 사용**
- `상태`: "시작 전" | "진행 중"
- `단계`: "🔴 이번주" | "🟡 다음주" | "⚪ 백로그"

**2-2) 프로젝트명 해석 (resolve)**

통합 태스크의 `프로젝트` relation 필드에서 추출한 URL 목록을 고유값으로 중복 제거한다.
`notion-fetch`로 각 프로젝트 페이지를 조회하여 프로젝트명(title)을 가져온다.

결과 예시:
```
프로젝트 URL: https://www.notion.so/32c19aac27e481b5b4b6f1048db43cff
→ 프로젝트명: "VRING:ON 제품 개선"
→ 소속 태스크: [{name: "Create/Variation 모델 학습 및 검증", ...}, ...]
```

**⚠️ 중요**: 스크럼 스레드는 **프로젝트** 단위로 생성된다. 하나의 프로젝트 아래에 내 태스크가 여러 개 있을 수 있다.

**2-3) 전체 프로젝트 DB 조회 (fallback)**

통합 태스크 DB에서 결과가 0건이면, 전체 프로젝트 DB를 fallback으로 쿼리한다:

```sql
SELECT url, "프로젝트명", "담당자(PL)", "상태"
FROM "collection://0ffbed16-cd0a-4381-b293-b3e630f1d06d"
WHERE "상태" = '[진행중] 프로젝트'
```

결과에서 `담당자(PL)` 필드에 `310d872b-594c-812e-b436-0002c6a46fa9`가 포함된 행만 필터링한다.
이 경우 "프로젝트명" 값을 프로젝트명으로 사용한다.

**2-4) 결과 정리**

최종 결과를 프로젝트별로 그룹핑한다:
```
[{
  project_name: "프로젝트명",       // 스크럼 스레드 매칭용
  project_url: "페이지URL",
  tasks: [{name, url, status, phase}, ...]
}]
```

### Step 3: 스크럼 메시지 찾기

`slack_read_channel`로 #스크럼 채널(`C09TF3XFJQ0`)의 최근 메시지를 조회한다 (limit 20).

필터링 조건:
- 발신자가 rbot(`U0AJT72CFLP`)인 메시지
- 당일 오전 9:25~9:35 KST 범위에 발송된 메시지
- 메시지 본문에 `Daily Scrum`이 포함된 메시지

각 메시지에서 프로젝트명을 추출한다. rbot 메시지 형식:
```
:thread: Daily Scrum · {프로젝트명}
영역: {영역} | Driver: {이름}
```
`Daily Scrum · ` 뒤의 텍스트가 프로젝트명이다.

### Step 4: 프로젝트-메시지 매칭

Step 2에서 조회한 **프로젝트명**(project_name)과 Step 3에서 추출한 스크럼 스레드의 프로젝트명을 비교한다.
- 부분 문자열 매칭: 프로젝트명이 스레드 제목에 포함되거나, 스레드 제목이 프로젝트명에 포함되는 경우 매칭
- 매칭된 메시지의 `message_ts`를 저장한다

**⚠️ 중요**: **프로젝트명**으로 매칭한다. 하나의 프로젝트에 내 태스크가 여러 개 있을 수 있으면, 그 태스크들의 내용을 모두 하나의 프로젝트 스크럼 스레드에 매칭된다.

매칭되지 않은 프로젝트는 스킵한다 (해당 프로젝트의 스크럼 메시지가 오늘 생성되지 않은 경우).

### Step 5: Daily DB 정보 수집 (핵심 데이터 소스)

`notion-query-data-sources`로 오늘과 조회 기간의 Daily 항목을 조회한다:

```sql
SELECT url, "Name", "date:Date:start"
FROM "collection://31119aac-27e4-8008-bb4b-000bd838f764"
WHERE "date:Date:start" IN ('{오늘}', '{조회_날짜1}', '{조회_날짜2}', ...)
ORDER BY "date:Date:start" DESC
```

각 Daily 페이지를 `notion-fetch`로 개별 조회하여 내용을 수집한다:

**조회 기간의 Daily 페이지에서 (복수 페이지 순회):**
- `[x]` 완료된 항목 → "완료된 업무" 목록으로 분류
- `[ ]` 미완료 항목 → 이월 항목으로 참고

**오늘 Daily 페이지에서:**
- `[ ]` 미완료 항목 → "진행할 업무" 목록으로 분류
- `[x]` 이미 완료된 항목 → "완료된 업무"에 포함 가능

Daily 항목 내에 프로젝트/태스크별 섹션 헤더가 있으면, 해당 프로젝트에 매핑한다.

**Daily DB 포맷:**

Daily 페이지는 `### todo` 섹션 아래에 작성된다. 두 가지 포맷이 혼재할 수 있다:

**포맷 A: 태그 있는 구조 (권장 포맷)**
```
### todo
- \[태스크명\]
	- [x] 완료 항목 (1탭 들여쓰기 = 하위 요소)
	- [ ] 미완료 항목
- \[다른 태스크명\]
	- [x] 완료 항목
### misc.
- 프로젝트에 속하지 않는 기타 업무
```

`- \[태그명\]`은 최상위 bullet이고, 그 아래 체크박스들은 탭 들여쓰기된 하위 요소이다.
`- \[태그명\]`과 Step 2에서 조회한 **태스크명**(tasks[].name)을 매칭한다.
부분 문자열 매칭: 태그명이 태스크명에 포함되거나, 태스크명이 태그명에 포함되는 경우 매칭한다.
(예: Daily의 `\[Create/Variation\]` 태그 ↔ 통합 태스크의 "Create/Variation 모델 학습 및 검증")

**포맷 B: 태그 없는 플랫 구조 (fallback)**
```
### todo
- [x] Create/Variation 피그마 슬라이더 라벨 반영
- [x] 그리드 오버뷰 레이아웃 수정
- [ ] trail_green_orange 재생성
```

태그가 없으면 **키워드 매칭**으로 각 항목을 태스크/프로젝트에 배정한다:
1. Step 2에서 조회한 태스크명 목록을 키워드로 사용
2. 각 체크박스 항목의 텍스트에 태스크명의 핵심 키워드가 포함되면 해당 프로젝트로 배정
3. 매칭되지 않는 항목은 `### misc.` 취급 (관련성이 있으면 포함 가능)

**포맷 판별**: `### todo` 섹션의 첫 번째 최상위 bullet이 `- \[...\]` 형식이면 포맷 A, 아니면 포맷 B.

`### misc.` 섹션의 항목도 관련성이 있으면 포함한다.

**서브페이지 추가 조회:**
각 Daily 페이지의 `notion-fetch` 결과에서 `<page url="...">` 태그가 있으면,
해당 URL로 `notion-fetch`를 추가 호출하여 서브페이지 내용도 수집한다.
서브페이지의 체크리스트 항목(`[x]`, `[ ]`)도 부모 페이지와 동일한 기준으로 분류한다.
단, 서브페이지가 3개를 초과하면 제목 기준으로 프로젝트 관련성이 높은 것만 선별하여 fetch한다.

Daily에서 수집한 항목은 태스크별로 분류되지만, 스크럼 메시지는 프로젝트 단위로 작성된다.
Step 2의 결과를 사용하여 태스크 → 프로젝트 매핑을 수행한다.
같은 프로젝트에 속하는 여러 태스크의 항목은 하나의 프로젝트 스크럼 메시지로 합친다.

### Step 6: 통합 태스크 DB 캐시 Diff (보조 데이터 소스)

이 단계는 Daily DB에서 특정 프로젝트 관련 항목이 **부족한 경우** 통합 태스크 DB의 상태 변화를 감지하여 보충한다.

**6-1) 전날 캐시 로드**

캐시 파일을 읽는다. 캐시 파일은 워크스페이스 내의 `.data/daily-scrum/cache.json`에 저장된다.
- 먼저 마운트된 워크스페이스 디렉토리를 동적으로 탐색한다: `/mnt/` 경로나 `ls /sessions/*/mnt/`를 사용하여 워크스페이스 마운트 경로를 찾는다.
- 캐시 파일 경로: `{workspace_mount}/.data/daily-scrum/cache.json`
- 파일이 없거나 읽기 실패 시: 캐시 없음으로 처리하고 6-2로 진행한다.
- 파일이 있으면 JSON을 파싱하여 전날 스냅샷을 메모리에 로드한다.

캐시 파일 구조:
```json
{
  "date": "2026-03-24",
  "schema_version": 4,
  "projects": {
    "프로젝트명": {
      "project_url": "https://www.notion.so/...",
      "tasks": [
        {"url": "task_page_url", "title": "태스크명", "status": "진행 중", "phase": "🔴 이번주"},
        {"url": "task_page_url", "title": "완료된 태스크", "status": "완료", "phase": "완료"}
      ]
    }
  }
}
```

**6-2) 현재 스냅샷 (Step 2 결과 재사용)**

Step 2에서 이미 조회한 통합 태스크 DB 결과를 캐시 스냅샷 형식으로 구성한다.
(추가 API 호출 불필요 — Step 2의 결과를 그대로 활용)

**6-3) Diff 비교 (캐시가 있는 경우만)**

전날 스냅샷과 오늘 스냅샷을 프로젝트별로 비교한다:

| 변화 유형 | 판별 기준 | 활용 |
|-----------|-----------|------|
| **새로 완료됨** | 전날 `status` != "완료" → 오늘 `status` == "완료" | "완료된 업무"에 추가 |
| **새로 진행 시작** | 전날 `status` == "시작 전" → 오늘 `status` == "진행 중" | "진행할 업무"에 추가 |
| **새로 추가된 태스크** | 전날에 없던 태스크 | 상태에 따라 분류 |
| **삭제됨** | 전날에 있었는데 오늘 없음 | 무시 |
| **변화 없음** | 상태 동일 | 무시 |

항목 매칭은 `url` 필드로 정확 매칭한다. URL이 변경된 경우 `title` 부분 문자열 비교로 fallback한다.

**중요**: Diff 결과는 Daily DB 데이터를 **보충**하는 용도이다.
- Daily DB에서 이미 충분한 항목이 있는 프로젝트는 Diff 결과를 사용하지 않는다.
- Daily DB에 항목이 부족한 프로젝트에 대해서만 Diff 결과를 추가한다.
- Diff에서 나온 항목은 표시 없이 자연스럽게 통합한다.

### Step 7: GitHub 커밋/PR 조회 (보조 데이터 소스)

Daily DB에 특정 프로젝트 관련 항목이 **부족한 경우에만** GitHub 데이터를 활용한다.

`gh` CLI로 해당 날짜 범위의 커밋/PR을 조회한다:
```bash
gh search commits --author=hayoung-jeremy --author=hayoung.kim --author=hossea3 \
  --owner=RebuilderAI --committer-date={시작일}..{종료일} --limit=20
gh search prs --author=hayoung-jeremy --author=hayoung.kim --author=hossea3 \
  --owner=RebuilderAI --updated={시작일}..{종료일} --limit=10
```
커밋 메시지와 PR 제목에서 해당 프로젝트와 관련된 항목을 추출하여 "완료된 업무"로 분류한다.
GitHub 데이터는 날짜가 정확하므로 신뢰할 수 있다.

`gh` CLI 미인증 시 이 단계를 건너뛴다.

### Step 8: 데이터 우선순위 정리

스크럼 메시지 작성 전, 수집한 데이터를 다음 우선순위로 정리한다:

**"완료된 업무" 소스 우선순위:**
1. Daily DB의 `[x]` 항목 (최우선 — 날짜가 정확)
2. GitHub 커밋/PR (날짜가 정확)
3. 통합 태스크 DB 캐시 Diff의 "새로 완료됨" 태스크 (상태 변화 기반)

**"진행할 업무" 소스 우선순위:**
1. Daily DB 오늘 페이지의 `[ ]` 항목 (최우선)
2. 통합 태스크 DB 캐시 Diff의 "새로 진행 시작" 태스크
3. 통합 태스크 DB에서 "진행 중" 상태 + "🔴 이번주" 단계인 태스크 (최후 fallback)

각 프로젝트별로, 상위 소스에서 충분한 항목(2개 이상)이 확보되면 하위 소스는 사용하지 않는다.

### Step 9: 스크럼 메시지 작성

매칭된 각 프로젝트별로 메시지를 구성한다. 형식:

```
완료된 업무:
• 완료 항목1
• 완료 항목2

진행할 업무:
• 진행 항목1
• 진행 항목2
```

**Slack 메시지 포맷 규칙:**
- Slack API(mrkdwn)에는 리스트 문법이 없다. 리스트 항목은 `•` (유니코드 bullet, U+2022)로 시작한다.
- `* ` (별표+공백)은 Slack UI 편집기에서만 bullet로 변환되고, API로 보내면 그대로 노출되므로 사용 금지.
- `- ` (하이픈)도 리스트로 변환되지 않으므로 사용 금지.
- 볼드: `*텍스트*`, 코드: `` `코드` ``, 링크: `<URL|텍스트>`
- "어제 완료", "오늘 진행" 표현 사용 금지. 반드시 "완료된 업무:", "진행할 업무:" 사용.

**태그 사용 조건** (해당되는 경우에만 추가):
- `이슈:` → 진행이 멈추거나 해결하지 못한 항목이 명확히 있을 때만 사용
- `도움 필요:` → Daily/프로젝트 페이지에서 누군가를 태그했거나, 누군가의 확인이 필요함을 명시한 경우만 사용
- `확인 필요:` → 위와 동일한 조건

태그가 불필요하면 사용하지 않는다. 억지로 만들지 않는다.

**작성 원칙:**
- 간결하게, 핵심만 기술
- 기술 용어/프로젝트 약어는 그대로 사용
- 과거 스레드 답변과 비슷한 톤 유지
- 불확실한 내용은 포함하지 않음

### Step 10: Slack DM으로 초안 전송

**2단계로 전송한다:**

**1) 본문 메시지** — `slack_send_message`로 김하영 DM(`U0AGKLZUCSY`)에 짧은 본문만 전송:
```
:memo: *Daily Scrum 초안* ({오늘 날짜})
```
전송 후 해당 메시지의 `message_ts`를 저장한다.

**2) 스레드 답장** — 위 메시지의 `message_ts`를 thread_ts로 사용하여, 프로젝트별 내용을 스레드 답장으로 전송:
```
*{프로젝트명}*
스레드: {스레드 퍼머링크}

{스크럼 메시지}
```
프로젝트가 여러 개면 하나의 스레드 답장에 모두 포함하거나, 프로젝트별로 별도 답장을 보낸다.

스레드 퍼머링크 형식: `https://rebuilderai.slack.com/archives/C09TF3XFJQ0/p{message_ts에서 점 제거}`

### Step 11: 캐시 저장 (내일을 위한 스냅샷)

Step 2의 통합 태스크 DB 결과를 캐시 파일에 저장한다.

**저장 절차:**
1. 오늘의 스냅샷 데이터를 JSON으로 구성한다:
```json
{
  "date": "{오늘 날짜}",
  "schema_version": 4,
  "projects": {
    "프로젝트명": {
      "project_url": "https://www.notion.so/...",
      "tasks": [
        {"url": "task_page_url", "title": "태스크명", "status": "진행 중", "phase": "🔴 이번주"},
        {"url": "task_page_url", "title": "완료된 태스크", "status": "완료", "phase": "완료"}
      ]
    }
  }
}
```
2. 캐시 파일 경로: `{workspace_mount}/.data/daily-scrum/cache.json`
3. 파일 저장 전에 `.data/daily-scrum/` 디렉토리가 없으면 생성한다 (`mkdir -p`).
4. 기존 내용을 덮어쓴다.
5. 저장 실패 시 에러를 DM으로 보고하되, 스크럼 전송 자체는 이미 완료되었으므로 태스크를 실패로 처리하지 않는다.
6. `schema_version`이 다른 기존 캐시는 무시하고 새로 생성한다 (이전 버전 → v4 마이그레이션 불필요).

### Step 12: 결과 보고

작성된 초안 내용과 대상 스레드 정보를 요약하여 알린다.
캐시 Diff에서 활용된 항목이 있으면 "(캐시 비교로 감지)" 표시를 결과 보고에만 포함한다 (Slack 메시지에는 포함하지 않음).


## 주의사항

- **Slack MCP와 Notion MCP 연결이 모두 필요하다.** 연결이 안 되어 있으면 DM으로 에러 메시지를 전송한다.
- 초안은 DM으로만 전송하고, **직접 스레드에 전송하지 않는다.**
- 매칭되는 스크럼 메시지가 없으면 "오늘은 담당 프로젝트의 스크럼 메시지가 아직 없습니다"라고 알린다.
- Daily DB에 오늘/어제 항목이 없을 수 있다. 이 경우 캐시 Diff와 GitHub fallback을 사용한다.
- 조회 기간은 직전 영업일부터 어제까지 모든 날짜를 포함한다. 월요일이면 금/토/일 3일치를 조회한다.
- Slack 메시지에서 em dash, 화살표 등 특수문자는 -, -> 등 ASCII로 대체한다. 특수문자가 invalid_blocks 에러를 유발할 수 있다.
- DM 메시지 본문에는 --- 구분선을 사용하지 않고, 빈 줄로 프로젝트 간 구분한다.
- 스레드 링크는 DM 본문이 아닌 스레드 답장으로 별도 전송한다.
- Slack MCP를 사용할 수 없는 경우, 스크럼 초안을 텍스트로 출력하고 "Slack MCP 연결 불가"라고 보고한다.
- **캐시 파일이 없는 첫 실행**: Diff 없이 기존 로직(Daily DB + GitHub)으로만 동작한다. 캐시 저장만 수행하여 다음 날부터 Diff가 활성화된다.
- **주말/연휴 후 캐시**: 금요일 캐시 → 월요일 비교 시 3일간의 변화가 한꺼번에 감지된다. 이는 정상 동작이다.
- **Notion 타임아웃**: notion-fetch, notion-query-data-sources 호출 시 타임아웃이 발생하면 최대 2회 재시도한다.
- **캐시 마운트 경로 동적 탐색**: 워크스페이스가 마운트된 경로는 실행 환경마다 다를 수 있다. 실행 시 항상 `/mnt/` 경로를 스캔하거나 `ls /sessions/*/mnt/`를 사용하여 활성 워크스페이스 마운트를 찾는다.

## 안전 원칙

- **다른 사람이 작성한 Notion 문서나 내용을 절대 임의로 수정/삭제하지 않는다.**
- 김하영이 관련된 작업(본인이 담당자인 태스크, 본인의 Daily 페이지 등)만 읽기/수정 대상으로 한다.
- 부득이하게 다른 사람의 콘텐츠를 수정/삭제해야 할 경우 반드시 DM으로 확인을 요청한다.

Notion 운영 방식이 변경될 수 있으므로, 하드코딩된 경로가 실패할 경우 동적으로 탐색한다:
1. **통합 태스크 DB 스키마 불일치 시**: `notion-fetch`로 `collection://db1ed8bf-4edf-4a87-b5bd-8774ce361a73`의 스키마를 조회하여 실제 컬럼명과 상태값을 확인한 뒤 재시도한다.
2. **전체 프로젝트 DB 스키마 불일치 시**: 동일하게 `collection://0ffbed16-cd0a-4381-b293-b3e630f1d06d`의 스키마를 조회하여 재시도한다.
3. **DB ID 자체가 변경된 경우**: 에러를 DM으로 보고하고, Daily DB + GitHub fallback으로 진행한다.
